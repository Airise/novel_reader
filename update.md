# 项目优化更新计划

结合当前项目的架构与已有实现，`update.md` 里的三个方向可以整理成一套更适合本项目落地的实施计划。整体目标是：**在不推翻现有 LangGraph + 混合检索 + rerank 架构的前提下，逐步升级为支持增量更新、检索降级和编辑辅助审校的中文长文本 Agent 系统**。

---

## 1. 问题1：增量更新（基于章节哈希的局部重建）

### 当前痛点

现在的索引流程更接近“整书重建”：

- 文本上传后重新切分
- 重新生成 chunk
- 重新写入 Qdrant
- 重新写入 Elasticsearch

这意味着只要小说某一章有变化，就可能要重建整本书，成本高、耗时长，也不利于后续做版本管理。

### 目标

将索引粒度从“整本书”提升到“章节”，做到：

- 只处理变化章节
- 未变化章节保持原索引不动
- 支持局部删除、局部重建、局部更新

### 实施方案

#### 1）章节级结构化存储

给文本引入章节级元数据：

- `chapter_id`：章节编号
- `chapter_title`：章节标题（可选）
- `chapter_hash`：章节内容哈希
- `raw_text`：章节原文

chunk 层再增加：

- `chunk_id`
- `chunk_hash`
- `chapter_id`
- `text`
- `uid`

#### 2）章节切分与 hash 对比

按章节拆分文本，可采用：

- 正则识别章节标题，如 `第[一二三四五六七八九十百千万0-9]+章`
- 若标题不规范，则结合空行和段落边界进行辅助切分

对每个章节计算 hash，保存到 `chapter_hashes.json`。上传新版本时，先对比新旧 hash，找出变化章节。

#### 3）局部删除与重建

对变化章节执行：

- 删除 Qdrant 中对应 `chapter_id` 的旧点
- 删除 Elasticsearch 中对应 `chapter_id` 的旧文档
- 重新切分该章节
- 重新生成 embedding 和 BM25 文档
- 更新 `novel_chunks.json` 与 `chapter_hashes.json`

#### 4）建议的落地顺序

- 第一阶段：先增加 `chapter_id` 和 `chapter_hash`
- 第二阶段：实现章节 hash 对比
- 第三阶段：实现 Qdrant / ES 的局部删除与重建
- 第四阶段：让前端“增量追加 / 覆盖重建”真正区分章节变化更新和全量更新

### 适配到当前项目的修改点

主要涉及：

- `indexing/` 下的切分逻辑
- `indexing/build_vector_store.py`
- `indexing/build_bm25_index.py`
- `index/novel_chunks.json` 的数据结构
- 新增 `index/chapter_hashes.json`
- 上传与重建索引流程

### 预期收益

- 改一个章节不再全量重建
- 索引更新速度明显提升
- 便于后续做版本对比与审校

## 2. 问题2：检索降级（Qdrant / ES 故障容错）

### 当前痛点

当前系统依赖双路检索：

- Qdrant 负责语义召回
- Elasticsearch 负责关键词召回

任一服务异常，都可能影响整个问答链路。

### 目标

在任一检索服务不可用时，系统自动切换到可用的检索器；在双路都不可用时，给出明确降级提示，而不是直接报错。

### 实施方案

#### 1）抽象统一检索接口

定义统一的检索抽象，例如：

- `search(query, top_k, filters=None)`

分别实现：

- `QdrantRetriever`
- `ESRetriever`

#### 2）引入健康状态管理

为每个检索器维护：

- `healthy`
- `failure_count`
- `last_failed_at`

建议策略：

- 连续失败 3 次 → 标记为 unhealthy
- unhealthy 状态下暂时跳过该检索器
- 每 30 秒做一次恢复探测

#### 3）自动降级策略

- Qdrant + ES 都正常：正常融合检索
- ES 故障：仅用 Qdrant 语义检索
- Qdrant 故障：仅用 ES BM25 检索
- 双方都故障：返回静态提示，并记录日志

#### 4）日志与监控

记录：

- 哪个检索器失败
- 失败次数
- 发生降级的时间
- 恢复的时间

### 适配到当前项目的修改点

主要涉及：

- `retrieval/hybrid_retriever.py`
- `retrieval/reranker.py`
- 新增检索状态管理模块
- 可能需要调整 Agent 节点里对检索失败的处理逻辑

### 预期收益

- 提高系统稳定性
- 防止单点故障导致整个问答不可用
- 更接近真实生产系统的容错设计

## 3. 问题3：从“小说问答”升级为“编辑辅助审校工具” + 追问拆解

### 当前痛点

当前系统更偏向“单轮问答 + 检索增强”。但在中文长文本或小说场景中，用户问题经常不是单点事实，而是：

- 追问
- 对比
- 矛盾检测
- 情节追踪
- 设定一致性检查

这些问题往往需要多步推理和多次检索。

### 目标

将系统升级为更偏“编辑辅助审校”的 Agent，能够：

- 识别问题类型
- 针对不同类型选择不同检索路径
- 自动拆解复杂追问
- 合并多轮证据后再生成回答

### 实施方案

#### 1）增加 Router 节点

在 LangGraph 中增加一个 Router 节点，先判断问题类型，例如：

- `fact`：简单事实
- `comparison`：对比
- `contradiction`：矛盾检测
- `tracing`：情节追踪
- `follow_up`：追问
- `search`：精确定位

#### 2）增加 SubQuestionGenerator 节点

对于复杂问题，尤其是 `tracing`、`follow_up`、`contradiction` 类型：

- 先让 LLM 生成子问题
- 最多控制在 3 个以内
- 避免过度拆解

例如：

用户问题：`张三后来怎么样了？`

可拆解为：

- 张三最后一次出现在哪一章
- 张三后续发生了什么
- 张三与其他角色关系是否变化

#### 3）修改 Executor

Executor 需要支持：

- 依次检索子问题
- 合并多个检索结果
- 去重
- 按章节聚合
- 再送入 rerank

#### 4）为矛盾检测设计专用路径

对于 `contradiction` 类型，建议：

- 先检索两个可能相关的章节或证据集合
- 再让模型对比回答
- 输出“是否矛盾 + 矛盾点 + 原文证据”

### 适配到当前项目的修改点

主要涉及：

- `agent/graph.py`
- `agent/nodes.py`
- `agent/state.py`
- `prompts/` 下的 planner / reflector prompt
- 检索结果合并与 trace 展示逻辑

### 预期收益

- 从单纯问答升级为智能审校辅助
- 更适合展示 LangGraph 的多步推理能力
- 对复杂追问和对比型问题支持更好

---

## 4. 推荐实施顺序

建议按下面的顺序推进：

### 第一阶段：章节级增量更新
优先把“局部重建”做出来，提升索引效率。

### 第二阶段：检索降级
增强系统稳定性和容错能力。

### 第三阶段：Router + 子问题拆解
把系统升级成真正的编辑辅助 Agent。

---

## 5. 建议的验收标准

### 阶段1验收
- 修改单章节后只重建受影响章节
- Qdrant 和 ES 的旧数据能正确删除
- 索引更新耗时显著下降

### 阶段2验收
- Qdrant 挂掉时系统能自动切到 ES
- ES 挂掉时系统能自动切到 Qdrant
- 双挂时有清晰降级提示

### 阶段3验收
- 系统能识别问题类型
- 复杂追问可以拆解成子问题
- 多子问题结果能够合并后再回答
- 矛盾检测类问题可走专门路径

---

## 6. 最终定位

在完成上述三步后，这个项目可以从一个“中文小说问答 RAG”升级为一个具备以下能力的系统：

- 支持增量索引维护
- 支持检索降级容错
- 支持复杂追问拆解
- 支持编辑辅助审校
- 适合中文长文本和网络文学场景

这会让项目从“能回答问题”升级为“能处理复杂文本理解和审校任务的 Agentic RAG 系统”。
