# AutoRAG 项目问题记录（开发过程）

> 目的：记录项目推进过程中出现的真实问题、根因、处理方式与经验，便于复盘与后续维护。  
> 范围：Week1（单次 RAG 基线）到 Week2（Agentic RAG 循环）。

---

## 1. 环境与依赖问题

### 1.1 虚拟环境创建了但安装没走 venv
- **现象**：`pip install -r requirements.txt` 报某些包无法安装（如 `qdrant-client` 版本不匹配）。
- **根因**：虽然创建了 `.venv`（Python 3.12），但安装时实际使用了系统 Python（3.7）。
- **处理**：
  - 先激活 `.venv`；
  - 用 `python -m pip install -r requirements.txt` 替代直接 `pip`。
- **经验**：Windows 下统一使用 `python -m pip`，避免解释器错位。

### 1.2 LangChain 生态依赖冲突
- **现象**：升级到 `langchain-openai==1.0.1` 后，`openai` 版本冲突。
- **根因**：`langchain-openai` 需要更高版本 `openai`。
- **处理**：将 `openai` 升级到兼容版本（`1.109.1`）。
- **经验**：LangChain 1.0 生态升级时，`langchain / langchain-openai / openai / langgraph` 要整体配套。

## 2. 脚本运行与导入问题

### 2.1 `No module named 'utils'`
- **现象**：直接运行 `python indexing/build_bm25_index.py` 时报导入失败。
- **根因**：以子目录脚本方式运行时，Python 模块搜索路径不含项目根目录。
- **处理**：在脚本中追加项目根路径（`sys.path.append(...)`）以兼容直接运行。
- **经验**：项目脚本要统一运行方式（推荐 `python -m ...`）或做路径兼容。

### 2.2 Prompt 文件编码异常（Windows）
- **现象**：`PromptTemplate.from_file(...)` 报 `UnicodeDecodeError: 'gbk' codec...`。
- **根因**：Windows 默认编码导致读取 UTF-8 prompt 文件失败。
- **处理**：改为显式 `open(..., encoding='utf-8')` 读取，再 `PromptTemplate.from_template(...)`。
- **经验**：跨平台项目中文 prompt 一律显式 UTF-8。

### 2.3 Prompt 变量转义问题
- **现象**：`Input to PromptTemplate is missing variables {'"action"'}`。
- **根因**：prompt 里 JSON 花括号未转义，被当成模板变量。
- **处理**：将 JSON 示例改为 `{{...}}`。
- **经验**：PromptTemplate 内凡是字面 `{}` 都必须双花括号转义。

## 3. 数据与预处理问题

### 3.1 小说文件编码不是 UTF-8
- **现象**：`preprocess.py` 读取 `novel.txt` 报 UTF-8 解码错误。
- **根因**：源文本可能为 UTF-16/GBK 等编码。
- **处理**：读取时做多编码回退尝试（`utf-8 / utf-8-sig / utf-16 / gb18030 / gbk`）。
- **经验**：中文语料入库前必须先做编码鲁棒读取。

### 3.2 仅按 `\n\n` 切分不稳定
- **现象**：chunk 过大/过小分布不均，影响召回。
- **根因**：原文本换行不规范，段落粒度不稳定。
- **处理**：
  - 文本归一化（换行、空白）；
  - 短段拼接；
  - 长段按句边界优先切分 + overlap 滑窗。
- **经验**：中文小说检索更适合“语义优先 + 长度约束”的 hybrid 切分。

### 3.3 末尾短块问题
- **现象**：质量报告中出现极短末尾 chunk。
- **处理**：增加“末尾回并”逻辑（过短则并入前一块）。

### 3.4 缺乏可观测性
- **现象**：很难判断切分是否合理。
- **处理**：新增 `chunk_quality_report.json`，记录长度统计、分布、样本。
- **经验**：切分质量必须量化，不应只靠主观感受。

## 4. 检索链路问题（Week1）

### 4.1 RRF 候选池过小
- **现象**：reranker 几乎不起作用。
- **根因**：RRF 后立即截断 top-3，再 rerank top-3。
- **处理**：引入 `FUSION_TOP_N`（如 30/60），先扩大候选再精排。

### 4.2 融合键使用文本导致冲突
- **现象**：不同段落文本相同时被误合并。
- **根因**：RRF 以文本做 key。
- **处理**：改为 `chunk_id` 融合，payload 存 `id + text`。

### 4.3 BM25 方案切换
- **现象**：希望去除 IK 插件依赖。
- **处理**：改为 `jieba` 预分词 + ES `standard` 分词器，新增 `text_jieba` 字段检索。
- **经验**：降低外部插件依赖可提升环境可移植性。

### 4.4 服务可用性异常
- **现象**：Qdrant/ES 任一未启动即崩。
- **处理**：
  - 初始化与查询处加 try/except；
  - 启动前健康检查（Qdrant、ES、DeepSeek）。

## 5. LLM 调用与兼容问题

### 5.1 OpenAI SDK 调用方式不兼容
- **现象**：旧式 `openai.ChatCompletion.create` 在新版本下不稳定。
- **处理**：迁移到 `OpenAI(...).chat.completions.create(...)`。

### 5.2 `llm.predict()` 弃用
- **现象**：LangChain 警告。
- **处理**：统一改为 `llm.invoke(...)`。

### 5.3 ChatOpenAI 导入路径演进
- **现象**：旧导入路径警告/兼容性差。
- **处理**：迁移到 `langchain_openai` 包。

## 6. Agentic RAG 循环问题（Week2）

### 6.1 规划输出不稳定
- **现象**：planner 偶发输出非 JSON 或 JSON 包裹解释文本。
- **处理**：增加稳健解析：先直 parse，再提取 `{...}` 片段解析，失败降级。

### 6.2 反思与校验标准不一致
- **现象**：reflector 判信息充分，但 verifier 判 FAIL，流程来回拉扯。
- **处理**：
  - 增加校验后反馈到流程控制；
  - 当校验 FAIL 时，反思节点可强制继续一轮检索。

### 6.3 循环空转与递归上限
- **现象**：触发 `GraphRecursionError`（默认递归限制 25）。
- **根因**：planner/refine 与 reflector/verifier 在边界条件下反复跳转。
- **处理**：
  - 增加 `max_steps`、`no_gain_rounds`、重复 query 检测；
  - verifier 路由收敛；
  - `graph.invoke(..., config={"recursion_limit": 60})` 提高上限。

### 6.4 “信息不足”偏多
- **现象**：明明有证据仍保守输出“信息不足”。
- **根因**：
  - 证据分散，生成阶段缺少结构化推理约束；
  - verifier 过严。
- **处理（泛化方向）**：
  - 生成改为“先事实、后推理、再结论”；
  - verifier 按通用三维评估（事实一致性 / 推理完整性 / 结论审慎性）。

## 7. 可观测性与调试能力建设

### 已增加
- `steps`：逐步轨迹；
- `retrieval_trace`：每轮 query、召回数、新增数、错误；
- `planner_raw / reflector_raw / verifier_raw`：原始模型输出；
- `errors`：错误列表；
- `test_agent.py --debug`：统一打印调试信息。

### 收益
- 能快速区分问题来自：
  - 检索没召回；
  - 规划 query 方向错误；
  - 反思误判；
  - 生成归纳失败；
  - 校验过严。

## 10. 本轮新增修复记录（去重复 query）

### 10.1 重复 query 仅按原字符串判断，拦截不充分
- **现象**：Agent 会重复发起“表面不同但语义等价”的 query（如仅空格、标点、大小写差异）。
- **根因**：重复检测仅做原字符串全等比较。
- **处理**：在 `agent/nodes.py` 新增 `_normalize_query()`：
  - 去空白
  - 统一小写
  - 去常见中英文标点
  - 再与历史 query 的归一化结果比较
- **结果**：减少重复检索与空转，提升有效步利用率。
- **后续可继续优化**：引入 embedding 相似度去重（语义级去重）。

### 10.2 校验失败后没有把“失败原因”反馈到下一轮规划
- **现象**：verifier 判 FAIL 后，下一轮 planner 仍可能重复旧方向检索。
- **根因**：校验结果只作为 PASS/FAIL 控制流，没有形成可执行的检索提示。
- **处理**：
  - 在 `AgentState` 新增 `verifier_feedback`；
  - verifier 解析第二行失败原因并写入 state；
  - planner 在下一轮把 `verifier_feedback` 拼接进提示上下文（作为“上轮校验反馈”）。
- **结果**：检索改写更有方向，减少“盲搜式”迭代。

### 10.3 校验反馈缺乏结构化标签，planner 难以稳定利用
- **现象**：同样的 FAIL 原因可能以不同自然语言表达，planner 难以形成稳定检索策略。
- **根因**：反馈仅有自由文本，没有统一缺陷类别。
- **处理**：
  - 在 `AgentState` 新增 `verifier_feedback_tag`；
  - verifier 输出第三行标签（`missing_entity / missing_relation / missing_time / unclear`）；
  - 若模型未给标准标签，则按关键词回退分类；
  - planner 将标签与文本反馈一起注入下一轮提示。
- **结果**：下一轮检索更可控，便于持续优化与统计分析。

### 10.4 连续同类 FAIL 时策略不切换，导致重复失败
- **现象**：连续多轮出现相同缺陷标签（如 `missing_relation`），但 query 仍在同一轨道微调，效果提升有限。
- **根因**：缺乏“失败模式检测 -> 策略切换”机制。
- **处理**：
  - 新增 `verifier_tag_history` 记录标签序列；
  - 当最近两轮标签一致且非 `unclear` 时，触发桥接式检索策略；
  - 自动构造短而聚焦的 bridge query（基于问题核心词 + 缺陷类型桥接词）；
  - 同时加入 query 压缩（长度/词数上限）避免 query 膨胀漂移。
- **结果**：减少“同类失败重复发生”，提高检索策略的自适应性与稳定性。

### 10.5 生成阶段出现“证据外信息”（幻觉）
- **现象**：答案中出现证据未明确提及的细节（例如“刺客”等），导致事实错误。
- **根因**：生成提示虽要求基于信息，但缺少“必须引用原文短引”和“无引文即信息不足”的硬约束；校验器对证据外实体/事件拦截不够强。
- **处理**：
  - 生成提示增加硬约束：每条关键事实必须附原文短引；若无可引文必须输出“信息不足”；
  - 校验提示增加硬校验：出现证据外人名/事件词，直接 FAIL，标签 `missing_entity`。
- **结果**：降低幻觉概率，提升答案可追溯性与可信度。

### 10.6 召回文本“词不达意/似是而非”
- **现象**：检索结果表面相关，但语义不够贴题，导致生成答案摇摆或跑偏。
- **根因**：rerank 后缺少最小相关性门槛，低质量候选仍会进入生成上下文。
- **处理**：
  - 在 `utils/config.py` 新增 `RERANK_MIN_SCORE`（默认 `0.15`）；
  - `retrieval/reranker.py` 增加阈值过滤：低于阈值直接丢弃；
  - 全部低于阈值时保底返回 top1，避免无上下文。
- **结果**：减少“奇怪片段”进入答案，提高上下文纯度与可读性。

### 10.7 前端运行慢、无法定位瓶颈
- **现象**：单问题耗时长，难判断卡在检索、LLM 还是循环控制。
- **根因**：缺乏节点级耗时观测；默认模式步数较多且包含 verifier 回环。
- **处理**：
  - 前端增加 `st.cache_resource` 友好调用层（缓存 `run_agent` 调用入口）；
  - 增加“快速模式”（`max_steps=3` + 跳过 verifier 二次回环）；
  - 在 `AgentState` 增加 `node_timings_ms`，记录 planner/executor/reflector/generator/verifier 节点耗时；
  - 前端展示总耗时与各节点平均耗时/调用次数。
- **结果**：可快速定位瓶颈并在体验与质量之间切换。

## 9. 复盘结论

- 项目从 Week1 到 Week2 的主要挑战不是“某个模型不行”，而是**工程闭环稳定性**：
  - 环境一致性、编码一致性、prompt 规范、循环收敛、可观测性。
- 当前系统已具备：
  - 混合检索 + 多步规划 + 反思 + 校验回路；
  - 基本可解释追踪能力。
- 下一阶段重点应转向：
  - 评测驱动优化（不是凭感觉调参数）；
  - 提升“复杂关系题”的通用推理稳定性。

## 11. 本轮新增修改记录（来源过滤多选 + 文档同步）

### 11.1 来源过滤由手动文本输入改为多选下拉
- **现象**：用户手动输入 `source_filter` 容易拼写错误，且不清楚当前有哪些可用来源。
- **根因**：前端仅提供文本框输入来源名，缺乏已入库来源枚举能力。
- **处理**：
  - 在 `chat_app.py` 新增 `get_available_sources()`：读取 `index/novel_chunks.json`，提取并去重 `source` 列表；
  - 将来源过滤控件改为 `st.multiselect`；
  - 增加帮助文案：“不选表示全库检索”。
- **结果**：来源过滤更稳定、可发现性更高，减少因拼写导致的“过滤后无结果”。

### 11.2 过滤参数传递方式与前端状态对齐
- **现象**：此前前端从文本框解析逗号分隔字符串，状态管理和可视化不直观。
- **处理**：
  - 改为直接使用多选结果 `selected_sources` 作为 `source_filter` 传入 `run_agent`；
  - 保持空列表语义为“不过滤，全库检索”。
- **结果**：参数语义更清晰，前后端交互更一致。

### 11.3 文档同步要求落地
- **需求**：用户要求“所有修改问题和优化都写入 `problem.md`”。
- **处理**：
  - 本次已将“来源过滤多选化”的问题、根因、处理、结果完整补充到 `problem.md`；
  - 后续每次功能修改将按同结构继续追加（现象/根因/处理/结果），保证可追溯。
- **结果**：开发变更与问题复盘形成持续文档闭环。
